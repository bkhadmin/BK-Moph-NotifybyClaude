Security Findings
🔴 ระดับ Critical — แก้ไขแล้วทั้งหมด
CRITICAL Session Cookie ไม่มี Secure Flag สำหรับ Public Flow
LINE Login cookie ถูก set โดยไม่มี secure=True ทำให้ cookie อาจส่งผ่าน HTTP ได้ แม้ว่า callback จะเข้าผ่าน Cloudflare HTTPS เสมอ
📁 web.py:1750 ✅ แก้ไขแล้ว: เพิ่ม LINE_LOGIN_COOKIE_SECURE=true (แยกจาก session cookie)
CRITICAL AccessLogMiddleware ไม่ทำงาน (Stub)
Middleware บันทึก Access Log เป็นโค้ดว่างเปล่า ทำให้ไม่มี audit trail ของ HTTP requests ซึ่งจำเป็นสำหรับระบบ Healthcare ที่ต้องสามารถ trace การเข้าถึงข้อมูลได้
📁 middleware.py:2 ✅ แก้ไขแล้ว: implement logging จริง (method, path, status, latency, IP)
CRITICAL Claim Signing Secret ใช้ค่า Default
claim_security.py มี fallback เป็น "change-me" ถ้าไม่ set env var ทำให้ผู้ไม่หวังดีสามารถปลอมลายเซ็น HMAC บน claim URL ได้
📁 claim_security.py:7 ✅ แก้ไขแล้ว: ยืนยันว่า production .env มี CLAIM_SIGNING_SECRET ที่แข็งแรงแล้ว
🟠 ระดับ High — แก้ไขแล้วทั้งหมด
HIGH Open Redirect ใน LINE Login Callback
/line/callback อนุญาตให้ redirect ไป http:// และ https:// ทุก domain ผ่าน query parameter next ทำให้ผู้ไม่หวังดีสามารถล่อให้ user กด LINE Login แล้ว redirect ออกไปยัง domain ปลอม (Phishing) ได้
📁 web.py:1748 ✅ แก้ไขแล้ว: อนุญาตเฉพาะ relative path ที่ขึ้นต้นด้วย /
HIGH Security Headers ไม่ครบ
SecurityHeadersMiddleware มีเพียง X-Frame-Options และ X-Content-Type-Options ขาด HSTS (Strict-Transport-Security), Referrer-Policy และ X-XSS-Protection ซึ่งจำเป็นสำหรับ production ที่รัน HTTPS
📁 middleware.py:5 ✅ แก้ไขแล้ว: เพิ่ม HSTS max-age=31536000, Referrer-Policy, X-XSS-Protection
HIGH Redis ไม่มี Connection Pool
Session store สร้าง Redis connection ใหม่ทุก request (redis.from_url()) ทำให้สิ้นเปลือง file descriptor และอาจเกิด connection exhaustion ภายใต้ traffic สูง
📁 session.py:3 ✅ แก้ไขแล้ว: ใช้ ConnectionPool shared ระดับ module
HIGH LINE Login Redirect URI ใช้ IP Hardcode
ค่า default ใน config.py ระบุ http://192.168.191.12:8012/line/callback ซึ่งเป็น HTTP และ IP address ไม่ใช่ domain จริง หากไม่ได้ override ด้วย .env อาจใช้งานไม่ได้ และ LINE จะปฏิเสธ redirect URI ที่ไม่ตรง
📁 config.py:58 ✅ ยืนยันแล้ว: production .env ตั้ง LINE_LOGIN_REDIRECT_URI เป็น HTTPS domain จริง
🟡 ระดับ Medium — ยอมรับความเสี่ยง (Accepted Risk)
MEDIUM CSRF Middleware ไม่ได้ Enforce (Stub)
CSRFMiddleware เป็นโค้ดว่าง form ใน admin มีเพียง login.html ที่ส่ง csrf_token อีก 22 form ยังไม่มี — อย่างไรก็ตาม admin panel อยู่ใน LAN เท่านั้น ไม่เปิดจากภายนอก และ public form 2 ตัวมีการป้องกันเฉพาะที่แน่นกว่า CSRF อยู่แล้ว (HMAC + LINE cookie)
📁 middleware.py:26 ⚠️ ยอมรับความเสี่ยง: ผลกระทบจริงต่ำมากเนื่องจาก architecture แยก LAN/Public ชัดเจน
MEDIUM CORS อนุญาต Methods และ Headers ทั้งหมด
allow_methods=["*"] และ allow_headers=["*"] กว้างเกินความจำเป็น อย่างไรก็ตาม ระบบไม่มี JavaScript cross-origin request เลย (ทั้งหมดเป็น form submit) และ ALLOWED_ORIGINS กำหนดเฉพาะ domain แล้ว ทำให้ CORS ไม่มีผลกระทบจริง
📁 main.py:50 ⚠️ ยอมรับความเสี่ยง: ไม่มี cross-origin JS call ในระบบ ความเสี่ยงเป็นศูนย์ในบริบทนี้

Existing Security Controls — Passed
  • ✅
    Bcrypt Password Hashing
    ใช้ bcrypt (cost factor ปกติ) สำหรับ hash รหัสผ่านทุกบัญชี ไม่เก็บ plaintext
  • ✅
    TOTP Two-Factor Authentication (2FA)
    Admin ทุกคนต้องผ่าน TOTP (Google Authenticator) หลังกรอก username/password
  • ✅
    HMAC-Signed Claim URL พร้อม Expiry
    ลิงก์รับเคสลงนามด้วย HMAC-SHA256 มีอายุ 1 ชั่วโมง ปลอม URL ไม่ได้
  • ✅
    LINE OAuth State Validation
    State parameter เก็บใน Redis 5 นาที ป้องกัน CSRF บน OAuth flow
  • ✅
    Server-Side Session (Redis)
    ไม่ใช้ JWT ใน cookie — session payload อยู่ใน Redis server ฝั่งเดียว revoke ได้ทันที
  • ✅
    IP Ban + Login Fail Limit
    Block อัตโนมัติหลัง 5 ครั้งล้มเหลว, ban IP ที่พยายาม brute force เกิน threshold
  • ✅
    SQL Guard (SELECT-Only)
    Query ที่ user กำหนดเองถูก whitelist ให้ SELECT เท่านั้น ป้องกัน DDL/DML injection
  • ✅
    Role-Based Access Control (RBAC)
    ทุก endpoint ตรวจสิทธิ์ตาม role ของผู้ใช้ ป้องกันการเข้าถึงฟีเจอร์เกินสิทธิ์

Future Recommendations
🔐
HTTPS สำหรับ Admin Panel (LAN)
ติดตั้ง self-signed certificate หรือ Let's Encrypt สำหรับ internal domain เช่น admin.bkhospital.local เพื่อให้เปิด SESSION_COOKIE_SECURE=true ได้ และป้องกัน session hijacking ใน LAN
🛡️
CSRF Token ครบทุก Form
เพิ่ม {% raw %}{{ csrf_token }}{% endraw %} ใน 22 form ที่เหลือใน admin templates แล้ว implement CSRFMiddleware จริง จะช่วยป้องกัน CSRF บน admin panel ได้สมบูรณ์
📋
Content Security Policy (CSP)
หลังจาก audit inline script/style ใน template แล้ว ให้เพิ่ม CSP header เพื่อป้องกัน XSS จาก injected content ควร audit ก่อนว่ามี inline script กี่ที่
🔄
Session Rotation หลัง Login สำเร็จ
สร้าง session ID ใหม่ทุกครั้งหลัง authenticate สำเร็จ ป้องกัน Session Fixation attack
⏱️
Rate Limiting บน Public Endpoints
เพิ่ม rate limit บน /alerts/claim และ /line/callback เพื่อป้องกัน automated abuse แม้จะมี HMAC guard อยู่แล้ว
